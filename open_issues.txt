ISSUE 3: Open/flat trade detection relied on flaky DOM-scraping
================================================================================
STATUS: Fixed - code committed in baf7fa6.

Symptom:
  User's framing: "fix this overcoming the dom limitation otherwise this
  whole project is scrap." getLatestTradeFromTester() in src/core/data.js
  scraped rendered text from the Strategy Tester's "List of Trades" panel -
  required scrolling a virtualized list, text-parsing heuristics, and a
  stability-polling loop (3-14s). Could time out under automation and had
  known false-positive/negative failure modes already documented in
  CLAUDE.md before this session.

Fix (src/core/data.js):
  - Added getStrategyPositionState(), which reads the strategy's own
    reportData() object directly off the chart's internal JS model
    (chart.model().model().dataSources() -> source matching /strategy/i ->
    .reportData()). performance.all.totalOpenTrades / performance.openPL are
    now the ground-truth open/flat signal - same object the Strategy Tester
    UI itself renders from, no DOM involved.
  - getLatestTradeFromTester() now tries this first. DOM scraping
    (getLatestTradeFromTesterDom, no longer exported) is used only as:
      (a) full fallback when no strategy source can be found/matched at all
      (b) a narrow, capped ~5s enrichment pass to recover an open position's
          entry timestamp specifically (reportData()'s filledOrders only
          carries a sequence index, not a real timestamp, for a still-open
          trade) - a DOM failure here never discards the open/flat status
          already established via the internal API.
  - Bug found + fixed during this work: the study-matching filter
    (`/strategy/i.test(name) || !!(s.reportData || s.performance ||
    s.ordersData)`) was too loose - those methods exist on the generic Study
    base class too (confirmed live: a "Dividends" study matched). Fixed in
    getStrategyPositionState, getStrategyResults, getTrades, and getEquity to
    match by name only.
  - Bug found + fixed during this work: a freshly-attached or just-switched
    strategy can return a mid-recompute snapshot if read once, immediately,
    with no settling check. Added readStableStrategyReportData(), which polls
    a few times (~350ms apart, bounded ~2s) and requires the trade-identity
    signature (open entry price/qty/id, or last closed trade's entry+exit
    price/time) to match twice in a row before trusting it. openPL is
    deliberately excluded from that signature since it fluctuates every tick
    for a genuinely open position.

Validated live:
  Ran a real scoped scan of the actual "Swing 15m" watchlist (all 13 symbols)
  and confirmed WTAI came back as OPEN/LONG, entry 42.43, unrealized
  +$113.40, source "strategy_internal_api" - matching the manually-confirmed
  ground truth.

NOT fully eliminated (by design, not an oversight):
  - The narrow ~5s DOM enrichment pass for entry-timestamp-on-open-trade
    still exists as a fallback. It can still fail/timeout; when it does, the
    system now correctly still reports OPEN/flat status and price, just
    without an entry timestamp for that one scan cycle - a much smaller
    failure than before (where DOM failure could erase the open status
    entirely).

Verify by:
  Check scan_results[].trade_source in the status JSON - should say
  "strategy_internal_api" for the common case, "strategy_internal_api+dom"
  only when the timestamp enrichment kicked in, and "strategy_tester_dom"
  only if no matching strategy source was found at all (would indicate a
  problem worth investigating, e.g. Issue 4 recurring).

Root cause (verified against status/latest-signal-status.json + live probe):

      TV UI renders as "Exit: Open")
    - rn (run-up = MFE) and dd (drawdown = MAE) are present for the open
      trade too
  So entry time, entry price, side, MFE, and MAE for open positions are ALL
  available from the internal API. The DOM enrichment pass is unnecessary
  and can be deleted, and the scanned_at fallback must be removed.
  (CLAUDE.md's claim that "only paired, closed trades carry real epoch-ms
  entry/exit times" is wrong and needs correcting as part of the fix.)

Fix plan:

ISSUE 10 (NEW, 2026-07-23): Excursion alerts never created ("Pending" forever);
no alert lifecycle (TV alerts never cleaned up after trade close)
================================================================================
STATUS: FIXED 2026-07-23 evening (uncommitted). Real create/delete schema
        captured and wired in; full lifecycle validated live end-to-end
        (create real alert -> confirm on account -> stage for cleanup ->
        drain -> confirm deleted from account).

SCHEMA CAPTURE (resolved the "STILL OPEN" item below):
  Root cause of the "invalid_request" error was never a transport/auth
  problem - it was captured correctly via CDP Network.requestWillBeSent
  while the user manually created + deleted one alert in the TV Desktop app.
  Two things were wrong in the old payload:
    1. The endpoint wants a {"payload": {...}} wrapper containing a
       `conditions` array (each condition: {type, frequency, series:
       [{type:"barset"},{type:"value",value}], resolution}), not the old
       flat {type, value} pair.
    2. The encoded symbol string (the '={"...":"..."}'-prefixed JSON blob)
       needs a `session` key ("extended") - omitting it was accepted
       client-side but rejected server-side.
  Endpoint: POST pricealerts.tradingview.com/create_alert, NO query params
  needed. Content-Type must stay text/plain;charset=UTF-8 (application/json
  still triggers the CORS-preflight failure noted below). Response on
  success: {"s":"ok","id":"...","r":{...,"alert_id":<number>,...}}.
  Delete: POST pricealerts.tradingview.com/delete_alerts, body
  {"payload":{"alert_ids":[<number>,...]}}, response {"s":"ok","r":null}.
  Only the "cross" condition type is confirmed live (fires once regardless
  of approach direction - fine for stop/target levels). "greater"/"less"
  are unverified best-effort guesses in toConditionType(), not load-bearing
  for this project since createExcursionAlerts only ever uses the default.

Changes shipped (src/core/alerts.js):
  - create() rewritten to the real payload shape above. Returns
    { success, alert_id, error, source: 'rest_api' } - alert_id is the real
    numeric TradingView id needed for later deletion.
  - deleteAlerts({ alert_ids }) now actually deletes via the REST endpoint
    (previously only delete_all: true existed, and only opened a context
    menu for manual confirmation - that DOM fallback is kept for delete_all).
  - src/tools/alerts.js (alert_delete) and src/cli/commands/alerts.js
    updated to accept alert_ids.
  - morning.js: new drainPendingAlertCleanup(baselinePath), called at the
    end of every runSignalJob() right after processLevelViolationsAndCleanup
    populates baseline.pending_alert_cleanup on trade close. Deletes each
    queued alert_ids batch via alerts.deleteAlerts(); entries that fail
    (e.g. transient network error) stay queued and retry next run instead
    of being dropped silently.

    excursion_alert entries have created:false since the feature was built.

Root causes found (src/core/alerts.js create()):
  1. Transport: POST to pricealerts.tradingview.com/create_alert with
     Content-Type application/json triggers a CORS preflight the server
     rejects -> fetch throws "Failed to fetch" on EVERY attempt. (GET
     list_alerts works because simple GETs skip preflight.) Confirmed:
     switching to a preflight-free content type gets HTTP 200.
  2. Payload schema: with transport fixed the server returns
     {"s":"error","err":{"code":"invalid_request"}} - the request shape in
     alerts.js was never correct. The real schema is being captured via CDP
     network sniffing while the user creates one alert manually in the TV
     UI (pending). NOTE for that fix: keep the preflight-free content type,
     and capture delete_alerts the same way (user deletes the test alert).
  3. Quota: account had 21 alerts vs MAX_ALERTS=20 in createExcursionAlerts;
     with 4 alerts/trade and 6+ open trades the design could not fit any
     plan tier comfortably.
